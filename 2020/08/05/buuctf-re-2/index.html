向前观察，发现有两组这样的操作：

将v26, v27, v28当作缓冲区，每次复制3个字符进行base64加密（这里的思路是顺理成章的，因为base64大致就是每次将3个字符切成4段按表查找的）。

那么每次复制的3个字符从哪里来？还需要向前寻找，发现根本没有对v20~v25的直接操作，那总不能是操作随机值吧？只好再看一看哪里可能改变了这些值。

注意到两个东西：那个定义成CHAR的String和调用的接口GetDlgItemTextA。

我这里先看的GetDlgItemTextA。

不认识，搜一下。

注意到第三个参数是用来接受对话框中的字符串的，这正好对应String。

那么String就有了解释：它是个数组，String连带着后面v19~v28很可能就是一个字符串。

代码中又没看到直接的flag相关的字符串，那么很有可能这个接受的String就是判断flag用的！

然后注意到一个比较字符串长度（可能之前就应该注意到）：

限制输入长度为8。

接下来对v7~v17挨个赋值，有了之前的经验，说不定这也是个字符串（或者数组），接下来就不得不面对之前的另一个函数sub_4010F0。

点进去简单浏览，只能猜测是对v7 ~ v17这个串做了某些操作。具体是什么操作还不急着看，先看看操作完之后用它干了什么。仔细看一下，其实只有最后MessageBoxA前面的判断用到了：

1.要判断输入字符串的第一个字符（String） = v7 + 34

2.要判断输入字符串的第二个字符（v19） = v11

3.要判断 v21 和 v14有一个数学关系，但注意到v21其实在后面另有限制，因此这个条件相当于没有。

我们突然发现：齐了！

String + v19 + v20v22 + v23v25 = 8

到此为止对输入的判断结束了。
